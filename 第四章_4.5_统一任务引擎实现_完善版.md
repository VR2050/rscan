## 4.5 统一任务引擎实现（完善版，侧重 Rust 实现）

统一任务引擎相关能力位于 `src/cores/engine/`，并与 CLI/TUI 的任务上下文协同工作。当前代码更准确地体现为“执行引擎 + 任务持久化 + 事件总线式记录”的组合，而非单一的中央调度器。

核心组成如下：

- 扫描执行引擎抽象：`src/cores/engine/engine_trait.rs`
- 异步连接扫描引擎：`src/cores/engine/async_engine/mod.rs`
- 原始报文扫描引擎：`src/cores/engine/raw_engine/*`
- 任务元数据与事件持久化：`src/cores/engine/task.rs`
- CLI 任务生命周期封装：`src/cli/common.rs`
- TUI 任务读取与展示：`src/tui/task_store.rs`

### 4.5.1 异步并发调度实现

当前引擎并发模型基于 Tokio，采用 `mpsc` 队列 + 多 worker 协作消费。

在异步连接扫描引擎中，任务提交与 worker 消费通过 `tokio::sync::mpsc` 串联：

```rust
// src/cores/engine/async_engine/mod.rs
let (task_sender, task_receiver) = mpsc::channel::<Task>(1000);
let task_receiver = Arc::new(tokio::sync::Mutex::new(task_receiver));

for _ in 0..num_workers {
    let task_receiver = task_receiver.clone();
    tokio::spawn(async move {
        loop {
            let task = {
                let mut receiver = task_receiver.lock().await;
                receiver.recv().await
            };
            ...
        }
    });
}
```

原始报文引擎同样采用队列驱动，并通过 `Semaphore` 控制在途任务规模，避免瞬时资源耗尽：

```rust
// src/cores/engine/raw_engine/mod.rs
let max_inflight = cfg.max_inflight.max(1);
let sem = Arc::new(Semaphore::new(max_inflight));
...
let permit = sem.acquire_owned().await?;
```

此外，`raw_engine` 还实现了目标级自适应节流（基于超时/错误反馈动态调整发送间隔），用于提升高压力场景稳定性。

说明：当前代码未体现“优先级队列（PriorityQueue）”和“Work Stealing”机制；调度策略主要是共享队列 + 多 worker + 信号量约束。

### 4.5.2 任务状态管理

任务状态定义在 `TaskStatus` 中，包含论文所述生命周期终态：

```rust
// src/cores/engine/task.rs
pub enum TaskStatus {
    Queued,
    Running,
    Succeeded,
    Failed,
    Canceled,
}
```

CLI 侧通过 `with_task(...)` 封装任务生命周期，统一写入开始/结束时间、状态与收尾日志：

```rust
// src/cli/common.rs
meta.status = TaskStatus::Running;
...
match result {
    Ok(_) => {
        meta.status = TaskStatus::Succeeded;
    }
    Err(_) => {
        meta.status = TaskStatus::Failed;
    }
}
meta.ended_at = Some(now_rfc3339());
```

该实现确保了状态变更与持久化的一致性：每次状态更新都落盘到 `meta.json`，并与事件流记录同步。

说明：`Canceled` 已在状态机模型中定义，但当前主执行路径以 `Running -> Succeeded/Failed` 为主，取消控制链路可作为后续强化点。

### 4.5.3 进度监控与事件记录

任务事件统一建模为 `TaskEvent`，事件类型覆盖日志、进度、指标与控制信号：

```rust
// src/cores/engine/task.rs
pub enum EventKind {
    Log,
    Progress,
    Metric,
    Control,
}
```

事件以 JSON Lines 追加写入 `events.jsonl`，实现可追溯执行轨迹：

```rust
// src/cores/engine/task.rs
append_jsonl_event(&self.task_dir.join("events.jsonl"), &ev)?;
```

进度更新通过 `TaskEventWriter::progress(...)` 上报，并同步刷新元数据中的 `progress` 字段。为降低噪声，代码对相邻重复进度进行桶级去重：

```rust
// src/cores/engine/task.rs
let bucket = (value.clamp(0.0, 100.0) * 10.0).round() as i64;
if self.last_progress_bucket == Some(bucket) {
    return Ok(());
}
```

该机制在长任务场景下可显著减少重复事件写入，提升事件文件可读性与 TUI 刷新效率。

### 4.5.4 结果持久化存储

任务数据采用“每任务独立目录”组织，默认结构与论文描述一致：

```text
workspace/
└── tasks/
    └── task-<id>/
        ├── meta.json
        ├── events.jsonl
        ├── stdout.log
        ├── stderr.log
        └── artifacts/
```

目录由 `ensure_task_dir(...)` 创建：

```rust
// src/cores/engine/task.rs
let dir = workspace.join("tasks").join(task_id);
fs::create_dir_all(&dir)?;
```

扫描输出通过 CLI 侧 `write_task_output(...)` 统一写入任务目录并注册产物：

```rust
// src/cli/common.rs
let path = write_output(...)?;
ctx.writer.register_artifact(path)?;
```

TUI 读取层从 `tasks/*/meta.json` 与 `events.jsonl` 加载任务列表、事件与产物预览，实现任务回放与结果检索：

```rust
// src/tui/task_store.rs
for ent in fs::read_dir(tasks_dir)? {
    let meta_path = task_dir.join("meta.json");
    ...
}
```

说明：当前“自动清理策略”在 reverse job 目录存在 `prune_jobs`，但 `workspace/tasks` 的统一保留策略尚未形成完整通用实现，可作为后续增强项。

### 4.5.5 小结与论文表述修订建议

结合当前 Rust 代码，建议 4.5 节采用“已实现能力 + 扩展方向”双层表述。

已实现：

- Tokio `mpsc` + worker 的异步任务执行框架
- `Semaphore` 在途任务限制与资源保护
- 任务状态持久化（`meta.json`）与事件流记录（`events.jsonl`）
- 统一任务目录下的日志与产物管理（stdout/stderr/artifacts）
- TUI 侧任务读取、事件展示与结果预览

扩展方向：

- 优先级调度（PriorityQueue）与抢占式调度策略
- Work Stealing 负载均衡模型
- 完整取消/暂停/恢复控制链路
- `workspace/tasks` 维度的通用保留与自动清理策略
- 更细粒度指标体系（吞吐、延迟分位、资源占用）

该写法能够保证论文与实现严格对齐，既体现工程完整性，也避免“描述超前于代码”的答辩风险。
