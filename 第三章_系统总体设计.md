# 第三章 系统总体设计

## 3.1 系统需求分析

### 3.1.1 功能需求分析

rscan_codex作为一款多合一安全扫描工具，需要满足以下核心功能需求：

**主机/端口扫描功能需求**：
- 支持TCP全连接扫描、UDP扫描、SYN半开扫描等多种扫描模式
- 支持ARP/ICMP主机发现，能够快速识别局域网内活跃主机
- 实现服务指纹识别，自动识别目标主机上运行的服务类型和版本
- 支持自定义端口范围、超时时间、重试次数等扫描参数
- 提供多种扫描配置文件（如low-noise、aggressive等）适应不同网络环境

**Web安全扫描功能需求**：
- 实现目录/路径爆破功能，支持自定义字典文件
- 支持Web Fuzz测试，能够对URL参数、HTTP头等进行变异测试
- 实现DNS子域名枚举，支持多种枚举策略组合
- 具备基础Web爬虫能力，能够自动发现网站链接和资源
- 支持robots.txt遵循和并发控制，避免对目标网站造成过大压力

**漏洞检测功能需求**：
- 构建模块化PoC执行引擎，支持YAML/JSON格式的规则定义
- 实现容器安全审计功能，能够检测Dockerfile和容器镜像中的安全风险
- 支持Web应用漏洞扫描，包括常见的注入、XSS、文件包含等漏洞类型
- 提供Fuzz攻击模板，便于用户自定义测试用例

**逆向分析功能需求**：
- 集成Ghidra反编译引擎，支持ELF、PE、APK等多种二进制格式
- 实现静态分析功能，能够提取函数、字符串、导入导出表等信息
- 支持任务化批处理，能够对多个二进制文件进行批量分析
- 提供轻量动态分析能力，通过strace抽样获取程序运行时行为
- 实现代码索引和搜索功能，便于在大型二进制文件中定位关键代码

**用户界面功能需求**：
- 提供命令行接口（CLI），支持完整的参数配置和脚本化使用
- 实现终端用户界面（TUI），提供直观的交互体验
- 深度集成Zellij终端复用器，支持原生Pane管理和多任务并行
- 支持智能命令补全，降低用户学习成本
- 提供实时进度反馈和结果展示，增强用户体验

### 3.1.2 性能需求分析

考虑到安全扫描工具的实际应用场景，系统需要满足以下性能需求：

**高并发性能**：
- 能够同时处理数千个并发网络连接
- 支持大规模端口扫描（1-65535全端口）在合理时间内完成
- Web扫描支持高并发HTTP请求，充分利用带宽资源
- 内存使用效率高，避免因并发数过高导致内存耗尽

**低延迟响应**：
- TUI界面响应时间不超过100ms
- 命令补全和输入反馈实时性高
- 扫描结果能够实时展示，无需等待任务完全结束

**资源效率**：
- CPU利用率合理，避免单核瓶颈
- 磁盘I/O优化，减少不必要的文件读写操作
- 网络带宽利用充分，支持自适应速率控制

**可扩展性**：
- 支持水平扩展，能够在多核CPU上充分发挥性能
- 模块化设计，便于添加新的扫描类型和分析功能
- 规则系统灵活，用户可以轻松扩展漏洞检测规则

### 3.1.3 安全性需求分析

作为安全工具本身，rscan_codex必须具备高度的安全性：

**内存安全**：
- 利用Rust的所有权系统，避免缓冲区溢出、空指针解引用等内存安全问题
- 安全处理来自网络的不可信数据，防止恶意输入导致程序崩溃或安全漏洞
- 在解析二进制文件时进行严格的边界检查和格式验证

**权限安全**：
- 最小权限原则，仅在必要时请求root权限（如SYN扫描）
- 权限分离，将高权限操作与普通操作隔离
- 提供明确的权限提示，让用户了解程序的权限需求

**数据安全**：
- 扫描结果和敏感信息的安全存储
- 支持结果加密和访问控制
- 避免在日志中泄露敏感信息

### 3.1.4 可扩展性需求分析

系统需要具备良好的可扩展性，以适应未来的发展需求：

**插件化架构**：
- 支持动态加载新的扫描模块和分析引擎
- 规则系统可扩展，用户可以自定义检测规则
- 接口标准化，便于第三方开发者贡献新功能

**配置灵活性**：
- 支持多种配置文件格式（JSON、YAML、TOML）
- 运行时参数可调，无需重新编译即可调整行为
- 提供丰富的API接口，便于与其他工具集成

**跨平台兼容**：
- 支持Linux、macOS、Windows等主流操作系统
- 在不同平台上保持一致的功能和用户体验
- 自动适配不同平台的特性和限制

## 3.2 系统架构设计

### 3.2.1 分层架构设计思想

rscan_codex采用分层模块化架构，将系统划分为五个主要层次：

**CLI层（Command Line Interface Layer）**：
- 负责命令行参数解析和用户输入处理
- 将用户命令转换为内部任务描述
- 提供命令帮助和参数验证功能

**核心引擎层（Core Engine Layer）**：
- 提供底层网络扫描和Web请求的核心能力
- 实现统一的任务调度和并发控制机制
- 定义标准化的数据模型和接口规范

**业务模块层（Business Module Layer）**：
- 实现具体的业务逻辑，包括主机扫描、Web扫描、漏洞检测、逆向分析等
- 各模块相对独立，通过核心引擎层进行协调
- 支持模块的独立开发和测试

**服务层（Service Layer）**：
- 提供通用的公共服务，如服务指纹识别、规则解析等
- 为业务模块提供共享的功能组件
- 实现跨模块的功能复用

**TUI层（Terminal User Interface Layer）**：
- 提供终端用户界面，包括Dashboard、任务管理、结果查看等功能
- 深度集成Zellij，实现原生Pane管理和多任务协作
- 处理用户交互事件和状态管理

这种分层架构具有以下优势：
1. **职责清晰**：每个层次都有明确的职责边界，便于开发和维护
2. **松耦合**：层次之间通过明确定义的接口交互，降低耦合度
3. **可测试性**：各层次可以独立进行单元测试和集成测试
4. **可扩展性**：新增功能可以在相应层次中实现，不影响其他部分

### 3.2.2 核心组件交互关系

系统的各个核心组件通过以下方式交互：

**任务生命周期管理**：
1. 用户通过CLI或TUI发起扫描任务
2. CLI/TUI层构建任务描述对象（TaskMeta）
3. 任务被提交到核心引擎层的任务队列
4. 核心引擎根据任务类型调度相应的业务模块
5. 业务模块执行具体扫描逻辑，通过事件系统报告进度
6. 扫描结果被持久化存储，并通过TUI层展示给用户

**数据流设计**：
- **输入数据流**：用户命令 → CLI解析 → 任务描述 → 业务模块执行
- **输出数据流**：扫描结果 → 事件系统 → 结果持久化 → TUI展示
- **状态数据流**：任务状态变更 → 状态更新事件 → TUI刷新

**控制流设计**：
- **同步控制**：用户交互、参数验证等同步操作
- **异步控制**：网络请求、文件I/O等异步操作通过Tokio运行时管理
- **并行控制**：CPU密集型任务通过Rayon进行并行处理

### 3.2.3 数据流与控制流设计

**数据流设计**采用发布-订阅模式：
- 任务执行过程中产生的事件（日志、进度、结果等）被发布到事件总线
- TUI层订阅相关事件，实时更新界面显示
- 结果持久化模块订阅结果事件，将数据写入磁盘

**控制流设计**采用命令模式：
- 用户的所有操作都被封装为命令对象
- 命令对象包含执行逻辑和撤销逻辑
- TUI层通过执行命令对象来处理用户交互

这种设计使得系统具有良好的可维护性和可扩展性，新的事件类型和命令类型可以很容易地添加到系统中。

## 3.3 模块化设计策略

### 3.3.1 CLI层设计

CLI层采用命令路由模式，将不同的子命令路由到相应的处理模块：

**命令结构**：
```
rscan [host|web|vuln|reverse] [subcommand] [options]
```

**路由机制**：
- `host` 命令路由到主机扫描模块
- `web` 命令路由到Web扫描模块  
- `vuln` 命令路由到漏洞检测模块
- `reverse` 命令路由到逆向分析模块

**参数处理**：
- 使用Clap库进行参数解析和验证
- 支持长选项（--option）和短选项（-o）两种格式
- 提供详细的帮助信息和使用示例

### 3.3.2 核心引擎层设计

核心引擎层是系统的中枢，提供统一的任务管理和并发控制：

**Engine Trait抽象**：
```rust
trait ScanEngine {
    async fn execute(&self, task: TaskMeta) -> Result<ScanResult>;
    fn supports_concurrent(&self) -> bool;
    fn max_concurrency(&self) -> usize;
}
```

**两种引擎实现**：
- **AsyncEngine**：基于Tokio的异步引擎，适用于I/O密集型任务
- **RawEngine**：基于原始套接字的引擎，适用于需要底层网络控制的任务

**任务调度器**：
- 维护任务队列和执行状态
- 根据任务类型和资源状况动态分配执行引擎
- 实现任务优先级和依赖管理

### 3.3.3 业务模块层设计

业务模块层采用策略模式，每个扫描类型都是一个独立的策略实现：

**Host模块**：
- `TcpScanner`：TCP全连接扫描策略
- `UdpScanner`：UDP扫描策略  
- `SynScanner`：SYN半开扫描策略
- `ArpScanner`：ARP主机发现策略

**Web模块**：
- `DirScanner`：目录爆破策略
- `FuzzScanner`：Fuzz测试策略
- `DnsScanner`：DNS子域名枚举策略
- `Crawler`：Web爬虫策略

**Vuln模块**：
- `PocExecutor`：PoC执行策略
- `ContainerAuditor`：容器审计策略
- `WebVulnScanner`：Web漏洞扫描策略

**Reverse模块**：
- `GhidraAnalyzer`：Ghidra分析策略
- `StaticAnalyzer`：静态分析策略
- `DynamicAnalyzer`：动态分析策略

每个策略都实现了统一的接口，可以通过配置文件动态选择和组合。

### 3.3.4 服务层设计

服务层提供跨模块共享的功能组件：

**ServiceProbe服务**：
- 解析nmap-style probe文件
- 实现服务指纹匹配算法
- 为所有扫描模块提供统一的指纹识别服务

**RuleParser服务**：
- 解析YAML/JSON格式的规则文件
- 验证规则语法正确性
- 为漏洞检测模块提供规则加载服务

**FileManager服务**：
- 管理扫描结果文件的存储和检索
- 实现文件去重和压缩
- 提供结果文件的访问接口

### 3.3.5 TUI层设计

TUI层采用MVC（Model-View-Controller）模式：

**Model（模型）**：
- `AppState`：应用程序全局状态
- `TaskStore`：任务数据存储
- `ProjectStore`：项目数据存储

**View（视图）**：
- `DashboardView`：仪表板视图
- `TaskListView`：任务列表视图
- `ResultView`：结果详情视图
- `ReverseWorkbenchView`：逆向工作区视图

**Controller（控制器）**：
- `InputHandler`：输入事件处理器
- `CommandExecutor`：命令执行器
- `ZellijIntegration`：Zellij集成控制器

TUI层通过观察者模式监听模型变化，自动更新视图显示。

## 3.4 关键数据结构设计

### 3.4.1 统一任务模型（Task Model）

系统采用统一的任务模型来描述所有类型的扫描任务：

**TaskMeta结构**：
```rust
pub struct TaskMeta {
    pub id: String,           // 任务唯一标识符
    pub kind: String,         // 任务类型（host/web/vuln/reverse）
    pub tags: Vec<String>,    // 自定义标签
    pub status: TaskStatus,   // 任务状态
    pub created_at: u64,      // 创建时间
    pub started_at: Option<u64>, // 开始时间
    pub ended_at: Option<u64>,   // 结束时间  
    pub progress: Option<f32>,   // 进度（0.0-100.0）
    pub note: Option<String>,     // 任务备注
    pub artifacts: Vec<PathBuf>,  // 产物文件路径
    pub logs: Vec<PathBuf>,       // 日志文件路径
    pub extra: Option<Value>,     // 额外字段（JSON值）
}
```

**TaskStatus枚举**：
```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum TaskStatus {
    Queued,      // 已排队
    Running,     // 运行中  
    Succeeded,   // 成功完成
    Failed,      // 执行失败
    Canceled,    // 已取消
}
```

这种统一的任务模型使得不同类型的任务可以共享相同的状态管理和持久化机制。

### 3.4.2 扫描结果数据结构（ScanResult）

扫描结果采用灵活的JSON结构，支持不同类型的结果格式：

**基本结构**：
```rust
pub struct ScanResult {
    pub task_id: String,      // 关联的任务ID
    pub target: String,       // 扫描目标
    pub findings: Vec<Finding>, // 发现结果列表
    pub metadata: Map<String, Value>, // 元数据
    pub raw_output: Option<String>,   // 原始输出
}
```

**Finding结构**：
```rust
pub struct Finding {
    pub id: String,           // 发现ID
    pub severity: Severity,   // 严重程度
    pub title: String,        // 标题
    pub description: String,  // 描述
    pub evidence: Value,      // 证据（JSON格式）
    pub remediation: Option<String>, // 修复建议
}
```

这种设计既保证了结果的一致性，又保留了足够的灵活性来适应不同类型的扫描结果。

### 3.4.3 任务状态机设计

任务状态机定义了任务在整个生命周期中的状态转换规则：

**状态转换图**：
```
Queued → Running → Succeeded
                ↘→ Failed  
                ↘→ Canceled
```

**状态转换规则**：
- 只有Queued状态的任务可以开始执行（转换为Running）
- Running状态的任务可以成功完成（Succeeded）、失败（Failed）或被取消（Canceled）
- Succeeded、Failed、Canceled状态是终态，不能再转换

**事件驱动的状态更新**：
- 任务开始执行时触发StartEvent
- 任务进度更新时触发ProgressEvent  
- 任务完成时触发CompleteEvent
- 用户取消时触发CancelEvent

状态机确保了任务状态的一致性和正确性。

### 3.4.4 事件系统设计

事件系统采用JSON Lines格式进行持久化存储，支持实时事件流处理：

**TaskEvent结构**：
```rust
pub struct TaskEvent {
    pub ts: u64,              // 时间戳
    pub level: String,        // 日志级别
    pub kind: EventKind,      // 事件类型
    pub message: Option<String>, // 消息文本
    pub data: Option<Value>,  // 附加数据
}
```

**EventKind枚举**：
```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum EventKind {
    Log,        // 日志事件
    Progress,   // 进度事件  
    Metric,     // 指标事件
    Control,    // 控制事件
}
```

事件系统为TUI提供了实时更新的能力，同时也为结果分析和调试提供了完整的执行轨迹。
