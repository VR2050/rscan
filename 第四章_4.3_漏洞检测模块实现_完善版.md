## 4.3 漏洞检测模块实现（完善版）

漏洞检测模块位于 `src/modules/vuln_check/`，当前实现由三类能力组成：

- 安全模板解析与校验：`safe_templates.rs`
- 多目标异步漏洞扫描执行：`scanner.rs`
- 容器配置审计（清单静态分析）：`container_audit.rs`

模块通过 CLI `vuln lint / vuln scan / vuln container-audit` 暴露统一入口，输出支持结构化结果（JSON）与可读化结果（pretty/raw）。

### 4.3.1 PoC 执行引擎设计

当前 PoC 执行引擎采用“安全模板子集（safe template subset）”思路：

1. 加载模板并解析为内部结构体 `SafeTemplate`。
2. 对每个 `target × template` 构建任务并发执行。
3. 按模板请求顺序发起 HTTP 请求，进行 matcher 判定。
4. 命中则输出 `VulnFinding`（模板、目标、方法、证据标签等）。

并发执行核心：

```rust
// src/modules/vuln_check/scanner.rs
let mut in_flight = stream::iter(work_items.into_iter().map(|(target, idx)| {
    let client = client.clone();
    let templates = Arc::clone(&templates);
    async move { execute_template_on_target(&client, &target, &templates[idx]).await }
}))
.buffer_unordered(cfg.concurrency.max(1));
```

单模板执行流程（请求构造、发送、匹配）：

```rust
// src/modules/vuln_check/scanner.rs
for raw_path in &req.paths {
    let rendered_path = render_runtime_vars(raw_path, &vars);
    let url = join_target_path(target, &rendered_path);
    let method = reqwest::Method::from_bytes(req.method.as_bytes()).unwrap_or(reqwest::Method::GET);
    let mut builder = client.request(method, &url);
    ...
    match builder.send().await {
        Ok(resp) => {
            let status = resp.status().as_u16();
            ...
            if eval_matcher(...) { ... }
        }
        Err(e) => out.errors.push(format!("{}: {}", url, e)),
    }
}
```

安全约束上，执行器内置方法白名单，避免危险或非预期请求类型：

```rust
// src/modules/vuln_check/scanner.rs
fn is_allowed_method(method: &str) -> bool {
    matches!(
        method.to_ascii_uppercase().as_str(),
        "GET" | "HEAD" | "POST" | "PUT" | "DELETE" | "PATCH" | "OPTIONS"
    )
}
```

说明：当前实现未看到“独立沙箱/系统调用隔离”执行机制，安全性主要来自“模板能力约束 + 方法白名单 + safe 模板子集”。

### 4.3.2 YAML/JSON 规则解析

规则解析器位于 `safe_templates.rs`，支持 `yaml/yml/json/txt`，并递归加载目录下模板文件：

```rust
// src/modules/vuln_check/safe_templates.rs
fn is_supported_template_file(path: &Path) -> bool {
    let ext = path.extension().and_then(|x| x.to_str()).unwrap_or_default();
    ext.eq_ignore_ascii_case("yaml")
        || ext.eq_ignore_ascii_case("yml")
        || ext.eq_ignore_ascii_case("json")
        || ext.eq_ignore_ascii_case("txt")
}
```

YAML/JSON 均反序列化为统一结构：

```rust
// src/modules/vuln_check/safe_templates.rs
pub struct SafeTemplate {
    pub id: String,
    pub info: TemplateInfo,
    pub requests: Vec<RequestDef>,
    pub source: PathBuf,
}
```

`RequestDef` 支持：方法、路径、头、Body、匹配器与提取器：

```rust
// src/modules/vuln_check/safe_templates.rs
pub struct RequestDef {
    pub method: String,
    pub paths: Vec<String>,
    pub headers: Vec<(String, String)>,
    pub body: Option<String>,
    pub extractors: Vec<ExtractorDef>,
    pub matchers: Vec<MatcherDef>,
    pub matchers_condition: String,
}
```

解析能力亮点：

- 兼容 `http:` 与 `requests:` 两种节点。
- 支持从 `raw` HTTP 报文抽取方法/路径/头/Body。
- 支持 `extractors`（regex）提取中间变量，并在后续请求中用 `{{var}}` 渲染。

说明：当前未实现论文原稿中提到的“条件分支、循环执行”等通用控制流语义；属于顺序请求 + 条件匹配模型。

### 4.3.3 容器安全审计实现

当前容器审计实现是“离线清单静态审计”，输入为 Kubernetes/容器编排清单（YAML/JSON），而非 Docker API 在线审计。

入口函数：

```rust
// src/modules/vuln_check/container_audit.rs
pub fn audit_container_manifests_from_path(
    path: impl AsRef<Path>,
) -> Result<ContainerAuditReport, RustpenError>
```

审计规则示例（高危配置）：

```rust
// src/modules/vuln_check/container_audit.rs
if get_bool(sc, "privileged") == Some(true) {
    push_finding(..., "high", "privileged-container", ...);
}
if get_bool(sc, "allowPrivilegeEscalation") == Some(true) {
    push_finding(..., "high", "allow-privilege-escalation", ...);
}
if let Some(caps) = get_value(sc, "capabilities")
    && let Some(add_caps) = get_value(caps, "add").and_then(|v| v.as_sequence())
    && add_caps.iter().any(|x| x.as_str().map(|s| s == "SYS_ADMIN").unwrap_or(false))
{
    push_finding(..., "high", "sys-admin-capability", ...);
}
```

模块还检查 `hostNetwork/hostPID/hostIPC`、`hostPath`、`runAsNonRoot`、`seccompProfile`、`readOnlyRootFilesystem` 等常见风险项，并输出结构化发现列表。

说明：当前未实现镜像层 CVE 匹配、恶意样本检测、Docker Runtime 在线配置拉取，这些应归类为后续扩展方向。

### 4.3.4 Web 漏洞检测模板

Web 漏洞检测采用模板驱动方式，模板由 `id/info/http(requests)/matchers/extractors` 描述。执行器支持常见 Nuclei 风格占位符：

```rust
// src/modules/vuln_check/scanner.rs
path.replace("{{BaseURL}}", target)
    .replace("{{RootURL}}", target)
    .replace("{{Hostname}}", &hostname)
    .replace("{{Host}}", &host)
    .replace("{{Port}}", &port)
```

匹配器类型当前主要包括：

- `status`：状态码匹配
- `word`：body/header 关键字匹配
- `header`：头部关键词匹配

```rust
// src/modules/vuln_check/scanner.rs
match m.kind.as_str() {
    "status" => m.status.contains(&status),
    "word" => { ... }
    "header" => { ... }
    _ => false,
}
```

因此，SQL 注入、XSS、命令注入、XXE、SSRF 等“漏洞类型”是否可检测，取决于模板 payload 与 matcher 规则本身；引擎提供的是通用执行与匹配框架。

CLI 层可按 `severity/tag` 过滤模板，提高任务针对性：

```rust
// src/cli/common.rs
pub(super) fn filter_vuln_templates(
    templates: Vec<SafeTemplate>,
    severities: &[String],
    tags: &[String],
) -> Vec<SafeTemplate> { ... }
```

### 4.3.5 小结与论文表述修订建议

结合当前代码，建议 4.3 节按“已实现能力 + 扩展方向”表达：

- 已实现：
  - 安全模板（YAML/JSON/TXT）解析与校验
  - 多目标并发模板扫描与结构化漏洞输出
  - Kubernetes/容器清单静态安全审计
  - 基于模板的 Web 漏洞检测框架（status/word/header matcher）
- 扩展方向：
  - PoC 执行隔离沙箱（进程级/容器级）
  - Docker API 与运行时配置在线审计
  - 镜像层 CVE 与组件 SBOM 关联分析
  - 更丰富 matcher（regex/dsl）与模板控制流（条件/循环）

该写法可确保论文描述与当前实现一致，降低答辩时“功能描述超前于代码实现”的风险。
